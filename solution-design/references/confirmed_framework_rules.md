# Confirmed Framework Rules

`confirmed_framework.md` is the second durable memory anchor. It is created only after the user confirms the technical route and framework.

## Required Sections

Use these sections unless the task needs a simpler structure:

1. Confirmed route
2. Confirmation basis
3. Rejected or comparison routes
4. System or algorithm modules
5. Inputs, outputs, and dependencies
6. Indicator system and verification logic
7. Figures, tables, and proposal evidence
8. Final proposal chapter framework
9. Open items that must be treated as assumptions

## Freezing The Framework

Treat this file as the final writing basis. The final `solution_design.md` should not introduce a different route, different module structure, fabricated performance claims, or new unsupported evidence.

If the user changes the route or framework after this file is created, update `confirmed_framework.md` before generating the final proposal.

## Proposal Framework

The chapter framework should be specific enough for final writing:

- include chapter names
- describe what each chapter must prove
- specify key module subsections
- list figures or tables that help explain the solution
- include page-count guidance only when useful

Keep the file concise. It should preserve decisions, not duplicate the full proposal.
