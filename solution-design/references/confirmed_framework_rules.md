# Confirmed Framework Rules

`confirmed_framework.md` is the second durable memory anchor. It is created only after the framework has been shown in chat and the user clearly approves the latest version.

Do not create `confirmed_framework.md` immediately after route selection. Route agreement starts framework drafting; framework approval creates the second anchor.

## Framework Must Appear In Chat First

The framework shown in chat should include:

1. chapter structure
2. section and subsection titles
3. what each chapter proves
4. selected route and comparison routes
5. system or algorithm modules
6. inputs, outputs, and dependencies
7. indicator system and verification logic
8. planned tables
9. figure placeholders and names
10. page allocation when useful

Ask whether the user approves this framework or wants changes. If the user requests changes, revise in chat and ask again.

## Required Anchor Sections

After approval, save these sections unless the task needs a simpler structure:

1. Confirmed route
2. Confirmation basis
3. Rejected or comparison routes
4. System or algorithm modules
5. Inputs, outputs, and dependencies
6. Indicator system and verification logic
7. Tables and figure placeholders
8. Final proposal chapter framework
9. Open items that must be treated as assumptions

## Freezing The Framework

Treat this file as the final writing basis. The final `solution_design.md` should not introduce a different route, different module structure, fabricated performance claims, or new unsupported evidence.

If the user changes the route or framework after this file is created, update the framework in chat first. Only update `confirmed_framework.md` after the user approves the revised framework.

## Proposal Framework

The chapter framework should be specific enough for final writing:

- include chapter names
- describe what each chapter must prove
- specify key module subsections
- list tables and figure placeholders that help explain the solution
- include page-count guidance only when useful

Keep the file concise. It should preserve approved decisions, not duplicate the full proposal.
